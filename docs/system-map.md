# JarHert: карта системы

JarHert — личный профиль Hermes. Это не набор отдельных ботов: Telegram —
единственный вход, а Personal OS SQLite — единственное хранилище личного
контекста. Внешние сервисы подключаются узкими адаптерами.

## Путь обычного сообщения

```text
Telegram
  -> Hermes gateway
  -> Codex backend + SOUL
  -> Tool Catalog / подходящий bundle
  -> Native MCP
  -> Personal OS или внешний адаптер
  -> итог в Telegram
```

1. **Telegram gateway** принимает текст, голос, callback кнопки и вложения.
   Он единственный владелец bot token и не конкурирует с отдельным старым
   gateway.
2. **Hermes + Codex backend** понимают запрос и формируют ответ. На живом
   профиле основной transport — `openai-codex`; API-модель остаётся fallback.
   Это не то же самое, что отдельный бинарник `codex` в shell.
3. **SOUL** задаёт русский тон, краткость, честные уточнения и формат результата.
   Он не хранит личные данные и не превращает стиль в имитацию другого человека.
4. **Tool Catalog** открывает не весь набор инструментов, а только нужный
   контекст: личное, планирование, исследование или код.
5. **Native MCP** вызывает типизированные операции. Модель предлагает действие,
   а инструменты проверяют входные данные, право владельца и idempotency key.
6. **Финальный ответ** возвращается в Telegram. Для долгих работ сообщение
   приходит из очереди после завершения, а не создаёт поток технических статусов.

## Когда пользователь просит что-то изменить

```text
Фраза пользователя
  -> список действий
  -> один preview
  -> одно подтверждение
  -> action plan
  -> Trello / Calendar / Personal OS
  -> один итог
```

- Напоминание, задача, встреча, заметка или несколько действий попадают в один
  action plan.
- До подтверждения внешняя система не меняется.
- После подтверждения plan выполняется один раз: повтор Telegram update не
  создаёт вторую карточку или встречу.
- Результат и external id остаются в SQLite для trace и компенсации.

## Данные и память

**Personal OS SQLite** хранит:

- заметки и базу знаний;
- проекты и project context;
- контакты, историю взаимодействий и обещания;
- напоминания, планы, очереди и результаты доставок;
- состояние мониторов и краткие события жизненного цикла.

FTS-поиск работает по заметкам локально. Сырые Telegram-экспорты не становятся
памятью автоматически: для них есть временное хранение и явное действие
пользователя.

## Внешние адаптеры

| Сервис | Для чего нужен | Как защищён |
|---|---|---|
| Trello | Задачи: создать, показать, перенести, закрыть | Один plan и подтверждение |
| Google Calendar | Встречи: создать, посмотреть, перенести, удалить | Один plan и подтверждение |
| GitHub public | Метаданные публичной репы, дерево и README | Без токена, короткий кэш |
| GitHub MCP | Private repos, PR, Actions, issues, security | Fine-grained token, read-only |
| Telegram MTProto | Экспорт текста из доступных владельцу чатов | Явный preview, TTL очистки |

Task Command Center остаётся внешним prerequisite: JarHert не знает его секреты
и не привязывает код к локальному пути Mac.

## Фоновые процессы

| Процесс | Что делает | Когда вызывает LLM |
|---|---|---|
| Reminders / outbox | Отправляет наступившие напоминания и сообщения | Никогда |
| Monitors | Сначала сравнивает hash и diff источника | Только при изменении |
| Daily Brief / Weekly Review | Собирает задачи, календарь, обещания | Только для короткой сводки, если включено |
| Memory consolidation | Объединяет подтверждённые факты | Ночным batch, не на каждый чат |
| Coding queue | Передаёт работу отдельному runner | По явной кодовой задаче |

## Код и Codex CLI

Разговорный Hermes уже использует Codex transport через OAuth-профиль. Прямой
Codex CLI нужен не для каждого сообщения, а для coding/research runner: он
получает изолированный workspace, репозиторий, задачу и критерий готовности,
после чего возвращает diff, тесты и итог. Он не должен иметь доступ к `.env`,
Docker socket, deploy или чужим каталогам.

На VDS runner не обязателен: сервер хранит очередь, а Mac или выделенный
изолированный runner выполняет код. Перед тем как считать Codex CLI рабочим,
нужен отдельный `scripts/coding_runner.py --check` с авторизованным CLI.

## Где смотреть состояние

- **Mini App → Память → Статус**: gateway, Trello, Calendar, GitHub, runner и
  backup.
- **Mini App → Память → Как работает**: короткая мобильная схема.
- **Telegram `/status`**: безопасная сводка без личного текста, токенов и сырых
  ответов провайдеров.
- **`/trace <id>`**: путь конкретного job через plan, действия и доставку.

## Границы

- JarHert не читает секреты из чата и не отправляет их в модели.
- GitHub MCP работает read-only.
- Отправка сообщения другому человеку, изменения Trello/Calendar и экспорт
  чатов требуют предусмотренного подтверждения.
- Deploy, merge, push и работа с host shell не являются обычными Telegram tools.

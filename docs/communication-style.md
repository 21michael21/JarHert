# Стиль общения JarHert

## Что внедрено

JarHert использует отдельный system prompt, собранный по измеримым признакам локального redacted-корпуса Telegram. Профиль влияет только на обычные AI-ответы. Парсинг команд, JSON extractor, monitors и tools продолжают работать со строгими техническими prompts.

Профиль доставляется нативно для каждого transport:

- OpenAI Responses API: поле `instructions`;
- OpenAI-compatible chat: сообщение с ролью `system`;
- Hermes CLI: `HERMES_EPHEMERAL_SYSTEM_PROMPT` только для дочернего процесса;
- Hermes HTTP: отдельное поле `system_prompt`.

Пользовательский текст не склеивается с системными правилами и не переписывается.

## На каких данных основан профиль

Локальный пакет содержит 1 388 уникальных redacted-постов: 1 318 train и 70 хронологических validation. Медианная длина — 383 символа. В корпусе измерены вопросы, условия, причинные связки, контраст и призывы к действию.

Raw-посты и training JSONL не лежат в JarHert и не отправляются модели при обычном запросе. В репозитории хранится только компактный производный профиль без цитат и идентификаторов источника.

## Почему сейчас нет fine-tune

В training JSONL нет user-сообщений: каждая запись содержит system + самостоятельный assistant-пост. Такой набор обучает форму публикации, но не реакцию на запрос, уточнение контекста и работу с tools. Fine-tune на нём может сделать ассистента похожим на генератор постов и ухудшить выполнение команд.

Поэтому текущая безопасная схема:

1. Данные задают наблюдаемые признаки стиля.
2. Runtime system prompt применяет их к ответу.
3. Детерминированный style eval ловит явный AI-slop.
4. Provider benchmark проверяет стиль на живых моделях.
5. Fine-tune откладывается до появления согласованного диалогового корпуса с user → assistant парами и отдельным holdout.

Это обучение поведения через conditioning и evaluation, а не изменение весов модели. «Внутреннее мышление» из постов не извлекается: мы задаём проверяемую политику ответа — понять задачу, отделить факт от допущения и дать следующий шаг.

## Настройка

По умолчанию используется `assistant/prompts/jarhert_communication_style.md`:

```env
AI_STYLE_ENABLED=true
AI_STYLE_PROMPT_PATH=
```

Для локального A/B можно отключить слой:

```env
AI_STYLE_ENABLED=false
```

Или указать свой локальный redacted prompt:

```env
AI_STYLE_PROMPT_PATH=/opt/jarhert-config/communication-style.md
```

Preflight проверяет существование и непустое содержимое файла. В лог выводится только hash-version профиля, не сам prompt.

## Проверка

```bash
.venv/bin/python scripts/eval_style.py
.venv/bin/python scripts/provider_benchmark.py --provider openai_cheap
scripts/local_check.sh
```

Style eval проверяет прямой ответ, честную неопределённость, ясный следующий шаг, пустые вступления, самореференсы ИИ, рекламный тон, крикливость и повтор предложений.

## Локальная дистилляция и A/B

Не отправляй корпус постов провайдеру и не подменяй им runtime на глаз. Длинные публикации полезны как
источник наблюдаемых правил, но не должны заставлять помощника писать простыни. Локальный дистиллятор
берёт только `assistant`-сообщения из уже redacted JSONL, ограничивает вклад каждой записи 500 символами и
создаёт короткий профиль в ignored `data/training/`.

```bash
.venv/bin/python scripts/distill_style_profile.py \
  /local/private/hermes_messages_train.jsonl \
  --output data/training/distilled_communication_style.md
```

Затем сравни текущий базовый режим и новый профиль на 100 одинаковых русских запросах. Внешней модели
передаются только эти тестовые запросы и два system prompt; исходные посты остаются локальными.

```bash
.venv/bin/python scripts/eval_style_ab.py \
  --candidate data/training/distilled_communication_style.md \
  --provider openai_cheap
```

Отчёт появляется в ignored `reports/style_ab/`. Кандидат можно включать через `AI_STYLE_PROMPT_PATH` только
если он выигрывает по средней оценке минимум на 3 пункта, не теряет число пройденных кейсов, набирает не
меньше 85/100 и проходит хотя бы 85% кейсов. Ничья и относительная победа слабого кандидата — не основание
менять живой профиль.

Дистиллированный профиль начинается со служебного комментария с метаданными:

```md
<!-- jarhert-style max_response_chars=320 -->
```

Он не уходит в system prompt, но задаёт кодовый лимит обычного ответа. Для явного «ответь коротко» лимит
ещё ниже. Runtime отдельно обрабатывает два однозначных случая: просьбу написать сообщение (возвращает
только готовый текст) и запрос о неизвестной причине без логов/данных (не пропускает догадки и хвостовой
вопрос). Это не применяется к action router, tools и JSON extraction.

## Условия для настоящего fine-tune

- явное право использовать каждый диалог;
- реальные user → assistant пары, а не придуманные prompts;
- удалённые токены, телефоны, имена и приватный контекст;
- отдельный chronological holdout;
- сравнение base vs conditioned vs fine-tuned на одних задачах;
- отсутствие регрессий JSON, tools, safety, factuality и latency;
- возможность мгновенно вернуться на базовую модель.
